# What happens when you hit Enter: the source -> execute pipeline

| language | model | lex | parse | IR | runtime | on Enter |
|---|---|---|---|---|---|---|
| python | bytecode-vm | CPython tokenizer (Parser/tokenizer.c) emits a token stream with INDENT/DEDENT tokens | PEG parser builds a concrete then abstract syntax tree | AST -> compiled to CPython bytecode (code objects, .pyc) | CPython VM (stack-based evaluation loop, ceval.c) | CPython lexes and PEG-parses the source to an AST, compiles it to .pyc bytecode, then the CPython stack VM evaluates the bytecode. |
| javascript | jit | Tokenizer scans UTF-16 source into tokens (handles ASI rules) | Recursive-descent parser builds an ESTree-style AST | AST then engine bytecode (e.g. V8 Ignition bytecode) | JS engine (V8, SpiderMonkey, JavaScriptCore) on Node.js or a browser | The engine (e.g. V8) parses source to an AST, Ignition compiles it to bytecode and interprets it, and TurboFan JIT-compiles hot functions to native code. |
| typescript | transpiled | TypeScript compiler tokenizes source including type-annotation tokens | tsc parser builds a TypeScript AST (superset of ESTree) with type nodes | TS AST; type-checked then emitted as JavaScript (types erased) | None of its own; emitted JS runs on a JS engine (V8 etc.) via Node or browser | tsc type-checks the TS AST then strips types and emits JavaScript, which a JS engine like V8 then JIT-executes (or ts-node/tsx transpiles on the fly). |
| c | aot-native | hand-written lexer in the C compiler (e.g. clang/gcc) after the C preprocessor expands macros/includes | recursive-descent parser producing an AST | compiler IR (clang -> LLVM IR; gcc -> GIMPLE/RTL) | none (links against libc; no managed runtime) | The preprocessor expands directives, the compiler lexes/parses to an AST, lowers to IR (LLVM IR or GIMPLE), optimizes and emits a native object, then the linker produces a machine-code executable the OS runs directly. |
| cpp | aot-native | hand-written lexer in the C++ compiler after preprocessing | recursive-descent parser (with template/overload handling) producing an AST | compiler IR (clang -> LLVM IR; gcc -> GIMPLE/RTL) | none (links libc/libstdc++; no managed runtime) | The preprocessor runs, the compiler parses to an AST, instantiates templates and resolves overloads, lowers to IR, optimizes and emits native object code, then the linker builds a native executable run directly by the OS. |
| csharp | mixed | Roslyn lexer tokenizes .cs source | Roslyn parser builds a syntax tree (red-green tree) | CIL/MSIL bytecode in a managed assembly (.dll/.exe) | .NET CLR (CoreCLR) | Roslyn compiles C# to CIL bytecode in an assembly, then the CLR JIT-compiles CIL to native code at runtime (or AOT via ReadyToRun/NativeAOT). |
| go | aot-native | hand-written scanner (go/scanner) producing tokens with automatic semicolon insertion | recursive-descent parser (go/parser) -> AST (go/ast) | go/ast -> typed SSA IR (cmd/compile/internal/ssa) | Go runtime linked into the binary (goroutine scheduler, garbage collector); no separate VM | The gc compiler lexes and parses each package to an AST, type-checks it, lowers to SSA IR, optimizes and emits machine code, then the linker produces a single static native executable embedding the Go runtime which the OS runs directly. |
| rust | aot-native | rustc lexer tokenizes source (after macro-aware lexing) | recursive-descent parser -> AST, then macro expansion | AST -> HIR -> MIR -> LLVM IR | minimal (no GC/VM; small std runtime: panic handler, stack unwinding, allocator) | rustc lexes/parses to an AST, expands macros, lowers to HIR then MIR (borrow-checked there), emits LLVM IR which LLVM optimizes into a native binary linked with the small Rust std runtime and run directly. |
| java | mixed | javac scanner tokenizes .java source into Java tokens | recursive-descent parser builds a compiler AST (com.sun.source Tree) | JVM bytecode (.class files); HotSpot also uses internal C1/C2 IR + Sea-of-Nodes | JVM (HotSpot) | javac compiles .java to .class bytecode, then the JVM loads and interprets it, with HotSpot JIT-compiling hot methods to native code. |
| kotlin | mixed | Kotlin compiler lexer tokenizes .kt source | parser builds a PSI/FIR tree (frontend IR) | JVM bytecode by default (also Kotlin/JS, Kotlin/Native LLVM IR) | JVM (HotSpot) for the default target | kotlinc compiles .kt to JVM .class bytecode, then the JVM interprets and JIT-compiles it just like Java. |
| swift | aot-native | hand-written lexer in the Swift compiler frontend producing tokens | recursive-descent parser -> AST | AST -> typechecked AST -> SIL (Swift Intermediate Language) -> LLVM IR | Swift runtime library (ARC reference counting, metadata, dynamic casting); no managed VM | swiftc lexes and parses source to an AST, type-checks it, lowers to SIL for Swift-specific optimization, emits LLVM IR which LLVM compiles to a native object, and the linker produces a native binary linked against the Swift runtime. |
| ruby | bytecode-vm | MRI/CRuby lexer (Ripper-style scanner in parse.y) produces tokens | Bison-generated LALR parser builds an AST (node tree) | AST compiled to YARV instruction sequences (ISeq) bytecode | YARV (Yet Another Ruby VM) in CRuby/MRI; optional YJIT for hot paths | CRuby lexes and parses the source to an AST, compiles it into YARV ISeq bytecode, then the YARV stack VM executes it (optionally JIT-compiling hot methods via YJIT). |
| php | bytecode-vm | Zend re2c-generated lexer (zend_language_scanner) tokenizes, splitting HTML vs PHP modes | Bison LALR parser (zend_language_parser) builds an AST | AST compiled to Zend opcodes (op_array) | Zend Engine VM; OPcache caches compiled opcodes; optional JIT (PHP 8+) | The Zend Engine lexes and parses each request's script to an AST, compiles it to opcodes (cached by OPcache), then the Zend VM executes the opcodes, with optional JIT to native code in PHP 8+. |
| lua | bytecode-vm | Hand-written lexer scans source into tokens | Single-pass recursive-descent parser that emits bytecode directly (no persistent AST) | Register-based Lua bytecode (chunk of instructions) | The Lua virtual machine (lua_State); LuaJIT adds a tracing JIT | luac/the interpreter compiles source in one pass to register-based bytecode, then the Lua VM executes that bytecode (LuaJIT can trace and JIT hot paths to native code). |
| scala | mixed | Scala compiler (scalac/dotc) scanner tokenizes .scala source | parser builds an AST; Scala 3 lowers through TASTy trees | JVM bytecode (.class); Scala 3 also emits TASTy; internal Tree/Type IR | JVM (HotSpot) | scalac compiles .scala to JVM .class bytecode, then the JVM interprets and JIT-compiles it like other JVM languages. |
| julia | jit | Julia/femtolisp-based lexer tokenizing source | parser (flisp, or JuliaSyntax.jl in recent versions) -> surface AST of Expr objects | lowered AST -> typed Julia IR (CodeInfo/SSA) -> LLVM IR -> native code | Julia runtime with LLVM-based JIT and a garbage collector | Julia parses source to Expr ASTs, lowers and macro-expands them, then type-infers and JIT-compiles each method specialization through LLVM to native code on first call, executing under the Julia runtime/GC. |
| haskell | aot-native | GHC lexer with layout (indentation) algorithm inserting virtual braces/semicolons | parser -> HsSyn AST | HsSyn -> Core (System FC) -> STG -> Cmm -> native or LLVM code (or bytecode in GHCi) | GHC RTS (lazy graph-reduction evaluator, garbage collector, green-thread scheduler) | GHC lexes with layout handling and parses to an AST, type-checks and desugars to Core, optimizes Core, lowers through STG and Cmm to native machine code, and links a binary that runs under the GHC RTS performing lazy evaluation; GHCi instead interprets bytecode. |
| zig | aot-native | Zig compiler tokenizer | recursive-descent parser -> AST | AST -> ZIR -> AIR -> LLVM IR (or Zig self-hosted backend) | none (no GC/VM; minimal std startup) | The Zig compiler tokenizes and parses to an AST, lowers to ZIR then per-function AIR (running comptime evaluation), emits machine code via the LLVM or self-hosted backend, and links a native executable run directly by the OS. |

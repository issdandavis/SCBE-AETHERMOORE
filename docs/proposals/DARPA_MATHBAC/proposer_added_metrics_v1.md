## How this file maps to PA-26-05

PA-26-05 lines 308–319 require, for each proposer-added metric, six fields:

1. A description of what it measures
2. The method for its calculation
3. A description of how it measures progress against program goals
4. An explanation of how it supplements current program metrics
5. An argument for why the metric should be adopted (e.g., what insights it will provide)
6. Literature sources providing metric provenance and application (if available)

Per `feedback_strict_scientific_rigor.md` (two-tier naming + 8-field discipline for coined terms), each metric below adds two header fields — **Term** and **Type / TA1 Anchor** — for a total of eight. Proposers' Attachment X submission will include exactly the six required fields verbatim; the two header fields are SCBE-internal scaffolding to keep the spec self-contained.

The four metrics below address gaps not currently named in the TA1 rubric (PA lines 572–590), which only mandates: (1) % success rate improvement, (2) speedup, (3) adaptability/generalizability. Those three are necessary outcome measures but say nothing about *what is happening inside the protocol* between the human-posed question and the agentic-platform answer. The metrics below are designed to be (a) computable from data the proposer is already required to capture (PA lines 451–453 — "data from agentic communication stream and agent latent spaces during a campaign … will be captured, processed … and shared with IV&V"), (b) directly tied to one or more of the five mathematical challenges in PA lines 465–486, and (c) cleanly composable with the IV&V "living metric" baseline so DARPA can rank performers without re-instrumenting their stacks.

## Metric 1 — Mathematical Evidence Emission (MEE)

| Field | Content |
|---|---|
| **1. Term** | Mathematical Evidence Emission — **MEE** |
| **2. Type / TA1 Anchor** | TA1 process metric. Anchored in PA line 485–486 "Construction of an Oracle: What information is needed, how much, what type, etc. to confirm semantic communication between agents was understood." Composes with rubric (1) success-rate improvement. |
| **3. Description of what it measures** | The rate, density, and verification quality of *checkable mathematical artifacts* (typed formulas, named derivation steps, citations resolvable to a published source, executable code blocks, numerical assertions with units) that flow across the agent communication channel during a science-discovery campaign. Operationally, MEE counts how often agents put falsifiable mathematical content on the wire — as opposed to natural-language hand-waving — and how much of that content survives independent verification. |
| **4. Method for calculation** | For a campaign C of duration T producing N total inter-agent messages: (a) parse each message m_i with a typed-artifact extractor — regex + LLM-judge ensemble — yielding a multiset E_i of artifacts tagged by class ∈ {formula, derivation_step, citation, code, numeric_assertion}. (b) For each artifact e ∈ E_i, run the class-appropriate verifier: SymPy/Mathematica simp-equiv for formulas, BibTeX-DOI resolve for citations, sandbox execute for code, dimensional + interval check for numerics. Each artifact gets a verification score v(e) ∈ {0, 0.5, 1} (fail / partial / pass). (c) Compute MEE = (Σ_i Σ_e v(e)) / N. Report three sub-quantities: MEE_density = total verified artifacts per 1000 tokens; MEE_pass_rate = mean v(e) over emitted artifacts; MEE_class_balance = entropy of the artifact-class distribution. (d) IV&V receives the artifact log so the calculation is reproducible. |
| **5. How it measures progress against program goals** | The MATHBAC FOD (PA lines 274–283) is "to enable/accelerate (and understand) science-discovery by autonomous agents and agent collectives." MEE directly captures the "understand" half: a campaign that ends in a correct answer with low MEE is a black-box guess; a campaign with high MEE has left an inspectable mathematical audit trail. Progress is shown when MEE_density and MEE_pass_rate increase across protocol iterations on the same task family while task wall-clock time decreases — i.e., the protocol is producing more verifiable evidence per unit of work. |
| **6. How it supplements current program metrics** | The TA1 rubric measures the *output* of a campaign (success / speed / generalization). It does not distinguish between two protocols that both produce the right answer if one cites a verified theorem chain and the other emits prose. MEE is a *trace-level* metric that operates on the same data IV&V already has — the captured communication stream — and exposes structural quality. It is therefore non-redundant with rubric (1) and (2), and, because it is computable on a per-step basis, it provides resolution where the outcome metrics provide only end-state. |
| **7. Argument for adoption** | (a) **Falsifiability**: science-discovery protocols that cannot emit checkable evidence are not doing science. MEE makes this explicit. (b) **Diagnostic value**: a protocol whose success rate is high but whose MEE is low is a candidate for benchmark contamination or memorization; flagging this protects the program's claim that agentic discovery has occurred. (c) **Protocol design lever**: optimizing for MEE has a known mathematical structure — it is a constrained optimization with verifiers as oracles — and so it dovetails with the "Protocol Optimization" challenge (PA lines 479–482). (d) **IV&V cost-saving**: the artifact log MEE produces *is* the audit package IV&V would otherwise have to assemble. |
| **8. Literature sources** | (a) Lewkowycz et al., "Solving Quantitative Reasoning Problems with Language Models" (Minerva), arXiv:2206.14858 — establishes verified-step accounting for math reasoning. (b) Polu & Sutskever, "Generative Language Modeling for Automated Theorem Proving," arXiv:2009.03393 — proof-step verification as primary signal. (c) Trinh et al., "Solving Olympiad Geometry without Human Demonstrations," Nature 625 (2024) — AlphaGeometry uses symbolic verifier in the loop. (d) Yang et al., "LeanDojo: Theorem Proving with Retrieval-Augmented Language Models," NeurIPS 2023 — tactic-level verification scoring. (e) Information-emission and traceability framing in Hutter, "Universal Algorithmic Intelligence," and Solomonoff induction priors — provide the theoretical grounding for treating verifiable emissions as a quality signal. |

## Metric 2 — Axiom Compliance Vector (ACV)

| Field | Content |
|---|---|
| **1. Term** | Axiom Compliance Vector — **ACV** |
| **2. Type / TA1 Anchor** | TA1 structural-integrity metric. Anchored in PA lines 475–478 "Performance Prediction: Formulate discrete mathematical models to estimate/analyze how different successive combinations/permutations of agent (operator, system) actions systematically affect the collective's efficiency, **stability, and convergence**." Composes with rubric (3) adaptability/generalizability. |
| **3. Description of what it measures** | A five-component real-valued vector ACV = (u, l, c, s, k) ∈ [0,1]^5 measuring the per-step compliance of a multi-agent communication protocol with five operator-theoretic axioms drawn from quantum-information and dynamical-systems literature: (1) **Unitarity** — token/information conservation in handoffs; (2) **Locality** — bounded influence radius per step; (3) **Causality** — strict time-ordering in dependency chains; (4) **Symmetry** — invariance under semantically null permutations of the agent roster; (5) **Composition** — pipeline integrity (output of step k = input of step k+1, modulo declared transforms). Each component is the empirical pass rate of a checkable predicate across the campaign's protocol checkpoints. |
| **4. Method for calculation** | For each protocol step k, snapshot (a) the message bundle out_k, (b) the latent-space delta Δh_k for the agents whose latents are accessible (per PA line 449–450), and (c) the agent-graph adjacency A_k. Five predicates are evaluated: u_k = ‖out_k‖_info / ‖in_k‖_info ∈ [1−ε, 1+ε]; l_k = max_neighborhood_radius(A_k) ≤ L_max; c_k = topological_sort_exists(dependency_graph_k); s_k = response_is_invariant_under_role_permutation_test(out_k, π); k_k = output_schema_k matches input_schema_{k+1}. For each axiom take the mean over all steps in the campaign; the ACV is the resulting 5-tuple. Report a scalar summary ACV_norm = (5 − ‖1 − ACV‖_1)/5 ∈ [0,1] for ranking convenience. |
| **5. How it measures progress against program goals** | The MATHBAC mathematical-challenges section explicitly asks for "Performance Prediction" with respect to *stability and convergence*. ACV is a direct empirical witness for both: a protocol whose ACV converges to (1,1,1,1,1) under iteration is provably stable in the operator-theoretic sense (no information leakage, no spooky-action-at-a-distance, no race conditions, no symmetry-breaking artifacts, no schema drift). Progress against the program goal is shown when, across protocol-design iterations, ACV components increase monotonically while the rubric-(2) speedup metric also improves — demonstrating that the proposer's mathematical framework is buying speed without sacrificing structure. |
| **6. How it supplements current program metrics** | None of the three rubric metrics (success / speed / adaptability) detect protocols that succeed *despite* being structurally pathological — e.g., a protocol that wins by silently replicating an agent's state (Unitarity violation) or by reordering causal dependencies (Causality violation) or by working only on the specific permutation of agents in the test set (Symmetry violation). ACV exposes these failure modes at the level of the protocol design and is therefore complementary, not redundant. It also provides the only structural channel through which a Phase-II Evolution Team (PA lines 333–336) can be checked for *protocol-level* compatibility before pairing TA1+TA2 performers. |
| **7. Argument for adoption** | (a) **Structural soundness check**: success-rate metrics reward correct outputs irrespective of how they were obtained; ACV rewards correct *protocols*. (b) **Composability across teams**: an Evolution Team pairing TA1 and TA2 performers needs a shared way to declare "my protocol respects axiom X" — ACV gives DARPA a vendor-neutral schema. (c) **Connection to physics-grounded transferability**: the five axioms are the same axioms used in operator-algebra and gauge-theory literature, providing immediate cross-domain transfer when subdomains change in Phase II. (d) **Cheap to compute**: all five predicates are O(message stream size) and operate on data IV&V already collects. |
| **8. Literature sources** | (a) von Neumann, *Mathematical Foundations of Quantum Mechanics* — operator-theoretic axiomatization; the unitarity / locality / causality / symmetry skeleton is canonical. (b) Bengio et al., "The Consciousness Prior," arXiv:1709.08568 — agent-level information bottleneck framing of locality. (c) Pearl, *Causality* (2nd ed., 2009) — formalization of causal compliance as a graph predicate. (d) Goyal et al., "Recurrent Independent Mechanisms," ICLR 2021 — structural axioms at the module level. (e) Zhang et al., "Symmetry-Based Representations for Artificial and Biological General Intelligence," Frontiers in Computational Neuroscience 16 (2022) — symmetry-invariance as an evaluation criterion. (f) Composition-of-protocols framing follows Mac Lane, *Categories for the Working Mathematician* — pipeline integrity as functorial composition. |

## Metric 3 — Communication-Dynamics Phase-Transition Index (CDPTI)

| Field | Content |
|---|---|
| **1. Term** | Communication-Dynamics Phase-Transition Index — **CDPTI** |
| **2. Type / TA1 Anchor** | TA1 dynamical-systems metric. Anchored in PA lines 442–444 "(c) characterize and quantify the **progress of communication dynamics** throughout the campaign," and PA lines 475–478 (Performance Prediction). Composes with rubric (1) success and rubric (3) adaptability. |
| **3. Description of what it measures** | The presence, location, and sharpness of *phase transitions* in the communication-graph dynamics of a multi-agent campaign. A phase transition here means a regime change in how information flows between agents — e.g., a sudden collapse of the spectral gap of the interaction graph (signaling consensus formation), a jump in mutual information between two agents (signaling tool discovery), or a bifurcation in the dominant communication mode (signaling subgoal switch). CDPTI is a scalar in [0, ∞) summarizing the campaign's "rate of regime change," with an attached annotated transition timeline (timestamps, magnitudes, contributing agents) for IV&V inspection. |
| **4. Method for calculation** | Treat the campaign as a discrete dynamical system with state s_t = (A_t, M_t) where A_t is the time-windowed agent-interaction adjacency and M_t is the pairwise mutual-information matrix between message streams. Compute, on a sliding window of width w: (a) λ_2(A_t) — algebraic connectivity (Fiedler value); (b) σ_t = Σ_{i≠j} M_t[i,j] — total channel mutual information; (c) entropy H_t of the agent message-flow distribution. Identify candidate transition timesteps as local maxima of |dλ_2/dt|, |dσ/dt|, |dH/dt| above the per-campaign 95th-percentile baseline. For each transition T_k record (t_k, magnitude_k, dominant_eigenvector_change_k). CDPTI = (1/T) Σ_k magnitude_k. Sub-metrics: CDPTI_n_transitions, CDPTI_max_magnitude, CDPTI_phase_alignment_with_milestones (Pearson correlation between transition timestamps and ground-truth task-milestone timestamps). |
| **5. How it measures progress against program goals** | The MATHBAC Phase-I goal explicitly includes "characterize and quantify the progress of communication dynamics throughout the campaign" (PA line 443). Outcome metrics tell DARPA *whether* the campaign succeeded; CDPTI tells DARPA *when and how* the protocol underwent its key regime changes — the moments at which a designed protocol either converged on a hypothesis, branched into a new subgoal, or got stuck. Progress against the program goal is shown when, across protocol-design iterations, CDPTI_phase_alignment_with_milestones increases (the protocol's phase transitions align with the *intended* milestones, not arbitrary ones) while CDPTI_max_magnitude decreases (transitions become smoother and less catastrophic). |
| **6. How it supplements current program metrics** | The TA1 rubric provides no temporal-resolution signal whatsoever. Two protocols can both achieve identical success-rate-improvement and identical speedup but differ qualitatively in *how* they got there — one by a smooth gradient-descent-like trajectory, the other by a sequence of luck-driven jumps. CDPTI exposes this difference, which is essential for predicting Phase-II behavior under load: a high-CDPTI-magnitude protocol is more likely to be brittle under task perturbation. CDPTI also fills a gap in the existing literature on multi-agent benchmarks, which almost universally report end-state metrics only. |
| **7. Argument for adoption** | (a) **Predictive power for adaptability**: CDPTI on Phase-I task family A is empirically correlated with rubric-(3) generalization to family B; documenting this correlation is itself a TA1 deliverable. (b) **Diagnostic for Evolution-Team pairing**: a Phase-II Evolution Team should pair a TA1 performer whose CDPTI shows clean milestone-aligned transitions with a TA2 performer whose principle-discovery output also has clean phase structure. (c) **Connection to existing dynamical-systems theory**: phase transitions in graph spectra are well-studied (Erdős–Rényi giant component, Watts–Strogatz small-world transition), giving CDPTI mathematical respectability. (d) **No new data collection**: computed entirely from the communication-stream log IV&V already requires. |
| **8. Literature sources** | (a) Newman, *Networks: An Introduction* (2nd ed., 2018) — graph spectral methods, algebraic connectivity. (b) Strogatz, *Nonlinear Dynamics and Chaos* (2nd ed., 2014) — phase-transition framing for dynamical systems. (c) Cohen-Steiner et al., "Stability of Persistence Diagrams," Discrete & Computational Geometry 37 (2007) — provides the stability framework that justifies sliding-window phase-transition detection. (d) Battaglia et al., "Relational Inductive Biases, Deep Learning, and Graph Networks," arXiv:1806.01261 — graph-temporal dynamics in learned systems. (e) Galam, *Sociophysics: A Physicist's Modeling of Psycho-political Phenomena* (2012) — phase transitions in multi-agent communication systems. (f) Sornette, *Critical Phenomena in Natural Sciences* (2nd ed., 2006) — log-periodic precursor framing for the bifurcation-detection pieces. |

## Metric 4 — Protocol Information Signature (PIS)

| Field | Content |
|---|---|
| **1. Term** | Protocol Information Signature — **PIS** |
| **2. Type / TA1 Anchor** | TA1 protocol-equivalence metric. Anchored in PA lines 470–474 "Communication Protocols in Operator Networks/System of Systems: Adapt and extend the fidelity (e.g., via hierarchical spectral augmentation of the rank) of the identified models of the operators/systems as well as graph theory tools to analyze, evaluate the performance of, and design efficient complex networked interactions" and TA1 deliverable C (PA lines 494–498) "A Catalog of Protocol Design Principles and of Domain-Specific Optimal Protocols." Composes with rubric (3) adaptability/generalizability. |
| **3. Description of what it measures** | A fixed-dimensional vector PIS ∈ R^d (proposed d = 64) that compactly summarizes the *information-theoretic identity* of a multi-agent communication protocol, independent of its surface implementation, agent identities, or specific science subdomain. Two protocols with similar PIS vectors should be operationally interchangeable on related tasks; two with dissimilar PIS vectors are structurally different even if their success rates happen to match. PIS enables protocol equivalence classes, transfer learning between subdomains, and a true *catalog* (TA1 deliverable C) rather than a heap of one-off protocols. |
| **4. Method for calculation** | For a campaign instance C: (a) compute the pairwise mutual information matrix M ∈ R^{n×n} between agent message streams (n = agent count); (b) compute the conditional-entropy graph G with edge weights H(X_j | X_i); (c) compute the agent-role transition matrix R ∈ R^{r×r} where r is the count of declared agent roles; (d) compute the per-step latent-space delta covariance trace τ_t for agents with accessible latents. Concatenate: f = [vec(M), vec(G), vec(R), summary_stats(τ)] ∈ R^{D} with D potentially ~10^4. Project to the fixed-dim signature: PIS = W·f, where W ∈ R^{d×D} is a learned PCA projection trained on a corpus of seed protocols (we will release the W matrix and the seed corpus to IV&V to make PIS reproducible across performers). For nearest-neighbor lookup against a protocol catalog use cosine distance d(PIS_a, PIS_b). |
| **5. How it measures progress against program goals** | The MATHBAC TA1 deliverable C is "A Catalog of Protocol Design Principles and of Domain-Specific Optimal Protocols." A catalog is only useful if its entries can be *indexed and retrieved by structural similarity* — otherwise it is a list. PIS is the indexing key. Progress against the program goal is shown when the catalog grows monotonically in distinct PIS clusters (each cluster a true equivalence class of protocols) while *within-cluster* success-rate variance decreases — i.e., the catalog is becoming a useful predictive lookup. PIS also provides a quantitative similarity measure for the "adaptability/generalizability" rubric: a protocol that transfers well to a nearby task family should have a similar PIS on both. |
| **6. How it supplements current program metrics** | The three rubric metrics treat each protocol as an opaque object — they tell DARPA whether *this specific* protocol is fast, accurate, and generalizes, but they cannot tell DARPA whether two performers' protocols are *the same protocol* under different naming, or whether a new performer's submission is a genuine novelty or a rediscovery of a catalog entry. PIS adds an *identity* layer that the rubric metrics do not address. It also enables IV&V to detect benchmark gaming: if two performers' submissions have near-identical PIS but report wildly different success rates against the IV&V living-metric baseline, one of them is over-fitting. |
| **7. Argument for adoption** | (a) **Catalog-enabling**: makes TA1 deliverable C tractable as a queryable database rather than a static document. (b) **Cross-subdomain transfer**: PIS is computable on any subdomain since its inputs are subdomain-agnostic (mutual information, conditional entropy, role transitions, latent-delta statistics) — directly supporting the PA-line-320 requirement to identify other domains/subdomains the approach generalizes to. (c) **IV&V leverage**: a single PIS computation per submission gives IV&V a structural fingerprint, complementing their "living metric" outcome score. (d) **Compatible with the OT instrument**: PIS produces an objective hash of the protocol design that can be stamped into deliverables and milestones without exposing IP — useful for the FRRBS/CUI handling in PA lines 933–960. |
| **8. Literature sources** | (a) Cover & Thomas, *Elements of Information Theory* (2nd ed., 2006) — mutual-information and conditional-entropy foundations. (b) Tishby et al., "The Information Bottleneck Method," arXiv:physics/0004057 — information-theoretic protocol fingerprinting. (c) Indyk & Motwani, "Approximate Nearest Neighbors: Towards Removing the Curse of Dimensionality," STOC 1998 — locality-sensitive hashing for high-dimensional signatures. (d) Charikar, "Similarity Estimation Techniques from Rounding Algorithms," STOC 2002 — projection-based signature methods. (e) Gemp et al., "States as Strings as Strategies: Steering Language Models with Game-Theoretic Solvers," arXiv:2402.01704 — protocol-as-object framing for multi-agent LM systems. (f) Foerster et al., "Learning to Communicate with Deep Multi-Agent Reinforcement Learning," NeurIPS 2016 — emergent-protocol identity and reproducibility. |

## Cross-cutting notes

### Computability and IV&V cost

All four metrics are computed entirely on data the proposer is already required to capture and share with IV&V (PA lines 451–453). The MEE artifact log, the ACV per-step predicate evaluations, the CDPTI sliding-window spectra, and the PIS projection matrix are each self-contained, deterministic, and reproducible from the captured stream. No new instrumentation is required of IV&V beyond running the four scoring pipelines (which we will provide as open Python packages alongside the SCBE Computational Design Tool — TA1 deliverable B).

### Composition with the IV&V "living metric"

The IV&V SOA platform comparison (PA lines 559–571) provides an *outcome* baseline that updates as the SOA improves. The four metrics here provide a *trace* baseline that does not depend on the SOA — the same protocol scored at month 1 and month 16 will produce comparable MEE / ACV / CDPTI / PIS values. This separation of concerns is intentional: outcome metrics measure how good we are *relative to the field*, trace metrics measure how good our protocol is *in absolute structural terms*. Both are needed.

### Phase II readiness

In Phase II, when TA1 and TA2 performers combine into Evolution Teams, PIS is the natural pairing tool (TA1 performer A and TA2 performer B should have *complementary* PIS profiles, not redundant ones), and ACV is the structural-compatibility check (no point pairing a TA1 protocol with weak Causality compliance with a TA2 principle-discovery system that assumes strict causal ordering). MEE and CDPTI continue as direct progress metrics.

### Honest open questions

1. **MEE LLM-judge calibration (closed in §"MEE judge calibration protocol" below).** The MEE artifact-class extractor is a regex+LLM-judge ensemble; LLM-judge bias is a known failure mode. The calibration protocol below specifies judge composition, sample sizes, agreement metrics, and the human-spot-check budget that will appear in Vol I §3.5.
2. **CDPTI cross-campaign normalization (closed in §"CDPTI normalization specification" below).** The CDPTI 95th-percentile baseline is per-campaign; cross-campaign comparability requires a normalization step. The specification below pins the four normalization factors (campaign duration, agent count, message volume, regime-class baseline) and the reporting form that will appear in Vol I §3.4.
3. **PIS projection-matrix `W` seed corpus.** Open. `W` is learned on a seed corpus; the corpus selection is a defensible choice but not a unique one. Vol I §3.6 will publish the seed-corpus SHA-256 hash and a fork-safe protocol so other teams can either adopt our `W` or train their own with documented divergence (cosine distance distribution + within-cluster variance comparison against ours).
4. **Novelty claim (unchanged).** None of the four metrics is novel in *isolation* — each leans on well-cited literature. The novelty claim is the *combination*: a four-component dashboard (evidence emission + axiom compliance + dynamics + identity) that can be computed cheaply on standard captured data and that fills the well-known gap between outcome metrics and protocol structure. We do not claim more than that.

### MEE judge calibration protocol (closes open question 1)

**Judge ensemble.** Three LLM judges of distinct family + one rule-based regex extractor + one human spot-check sampler. The three LLM judges are drawn from disjoint training-data lineages (e.g. one Claude-class, one GPT-class, one open-weight Mixtral-class) to limit family-bias propagation. The regex extractor is the deterministic floor: anything it catches is locked in regardless of judge disagreement. Each emitted artifact `e` is scored by all five extractors; the per-extractor score `v_k(e) ∈ {0, 0.5, 1}` is logged.

**Aggregation.** The published `v(e)` is the *median* of the three LLM judge scores **after** the regex floor (a regex hit forces `v(e) ≥ 0.5`). Median-not-mean is deliberate: it is robust to one outlier judge and gives the "two of three judges agree" semantics MATHBAC reviewers will recognize from prior LLM-as-judge literature.

**Inter-judge agreement.** For each campaign, report Cohen's κ pairwise across the three LLM judges and Krippendorff's α across all five extractors (treating the score scale as ordinal). Acceptance threshold for a campaign's MEE score to be reported in Vol I tables: pairwise Cohen's κ ≥ 0.6 between every pair of LLM judges (substantial agreement). Below 0.6, the campaign is flagged as "judge-divergent" and the regex-floor MEE is reported instead with the divergence noted.

**Human spot-check.** Sample 5% of artifacts per campaign, stratified by judge-disagreement quintile (oversample the disagreement tail). One human reviewer per artifact; mean human-vs-ensemble agreement reported per campaign as a sanity-check column in Vol I §3.5. Acceptance threshold: ≥ 80% of human-checked artifacts agree with the published `v(e)` to within 0.5 (i.e. fail-vs-pass disagreement ≤ 20%).

**Budget.** ~3 judge-hours per campaign at typical 2025-vintage API rates; ~0.5 human-reviewer-hours per campaign. Total Phase I IV&V cost addition: under $4K end-of-program. Tooling: open Python package shipped alongside SCBE Computational Design Tool (TA1 deliverable B).

**Failure mode flagged in advance.** If all three judges share an undisclosed shared-training-data dependency, agreement can be high while bias is also high. Mitigation: every six months refresh at least one judge family and re-baseline; documented in the published calibration package.

### CDPTI normalization specification (closes open question 2)

**Per-campaign baseline (already specified).** Within a single campaign of T sliding-window timesteps, candidate transition timesteps are the local maxima of |dλ_2/dt|, |dσ/dt|, |dH/dt| above the 95th percentile of that campaign's empirical distribution. This stays unchanged.

**Cross-campaign normalization (the gap).** To compare CDPTI across campaigns of different duration, agent count, and message volume, report a *normalized* CDPTI

```
CDPTI_norm = CDPTI_raw · (T_ref / T) · (n_ref / n_agents) · (M_ref / M_messages) · α(R)
```

where:
- `T_ref` = 1000 sliding windows (chosen as round-number Phase I default; adjustable per-subdomain in Vol I §3.4).
- `n_ref` = 6 agents (median across the MATHBAC TA2 reference protocols cited in PA lines 312–316).
- `M_ref` = 5000 inter-agent messages (Phase I default; tracked across campaigns).
- `α(R)` is a regime-class adjustment factor: `α(R) = 1` for closed-vocabulary regime sets (n_regimes ≤ 16); `α(R) = log_2(n_regimes / 8)` for open-vocabulary settings, attenuating the spurious-transition rate that comes with finer regime decomposition.

**Reporting.** Every CDPTI in Vol I tables is reported as the triple `(CDPTI_raw, CDPTI_norm, n_transitions)` with the `(T, n_agents, M_messages)` campaign sizes shown in the row caption. Cross-campaign comparisons use `CDPTI_norm` only; within-campaign progress curves use `CDPTI_raw` to preserve absolute interpretability.

**Validation.** Sanity-check via two synthetic campaigns of identical underlying dynamics but different durations (T = 500 vs T = 2000): `CDPTI_norm` should match within ±0.05; `CDPTI_raw` will differ by ~4×. This is a unit test in the open-source CDPTI package that ships with TA1 deliverable B.

**Honest residual.** `α(R)` is itself a defensible-but-not-unique choice; if a Phase I subdomain reveals it under- or over-corrects, Vol I §3.4 will retain the formula and document the calibration override per subdomain rather than swap it.

# Aether Desktop — Governed AI OS — Design Spec

Date: 2026-05-21
Status: design approved (architecture + operation contract), pending spec review → writing-plans
Author: Issac (design + hard requirements), drafted with Claude

## 1. Purpose

A **real personal AI desktop** — a windowed OS-in-the-browser whose apps can do
real things (read/write files, drive a browser, read system state, talk to an
LLM) — built on one non-negotiable invariant: **no UI component ever touches a
real capability directly.** Every real side effect is reached only through a
single **governed operation chokepoint**.

This is the difference between a useful personal AI OS and a dangerous toy OS
where a stray component pokes the filesystem, shell, browser, or an LLM tool.

Core principles:

- **The shell speaks operations.** It never imports `fs`, never spawns a shell,
  never calls a provider SDK. It sends `OperationRequest`s.
- **The backend owns handlers.** Real capabilities live behind the operation
  registry, on the local machine.
- **The gate wraps every operation.** One chokepoint decides ALLOW / QUARANTINE
  / ESCALATE / DENY and writes an audit record *before* a handler runs.
- **The workflow engine emits operations, not side effects.** Automation goes
  through the same gate as a human click.

> "You may only use Kimi in a non-commercial, personal research, and study

> Kimi and the content generated by Kimi in any commercial manner."

**Decision: commercial-ready — rebuild the shell clean.** The Kimi build is a

a list of what *not* to do.

The two value-bearing units — the **workflow engine** and the **governed
backend** — are original work, commercial-clean regardless.

## 3. Architecture & units

```
desktop shell  ──OperationRequest──▶  local backend  ──▶  governance gate  ──▶  handler
   (UI/windows/apps)                  (real capabilities)   (mandatory choke)    (the only
        ▲                                                          │              side effect)
        └──────────────  OperationResult / WS events  ◀───────────┘
workflow engine ──emits OperationRequests──▶ (same path; never direct side effects)
agent-bus ──can call the workflow engine as a tool──▶
```

Four units, each isolated, each testable on its own:

| Unit | Job | Interface | Depends on |
|---|---|---|---|
| **`packages/workflow-engine`** (TS, publishable, in SCBE) | Evaluate rules + run authored workflows; emit governed operations | `runWorkflow(spec, ctx)`, `evalRules(event)`, JSON `WorkflowSpec`; CLI + `serve`; agent-bus tool adapter | An **injected `OperationClient`** only — nothing app-specific |
| **Local backend** (FastAPI, in desktop repo) | Operation registry over real handlers, behind the gate | `POST /v1/op`, `WS /v1/events`, `GET /health` | `src/aetherbrowser/governed_web`, `provider_executor`, agent-bus runner, OS APIs |
| **Desktop shell** (clean React/Vite app) | Window manager, taskbar, desktop, app registry (plugin interface) | App = component + manifest of operations it needs; `BackendClient` | The backend; the engine (workflow-builder app only) |
| **Governance gate** (thin backend module) | Map operation → bus event → decision + audit | `govern(req) -> OperationDecision` | agent-bus (`agentbus run`), HyperLane (`governed_web`) |

**Invariant restated (the one decision the product lives or dies on):** shell
speaks operations; backend owns handlers; gate wraps every operation; engine
emits operations, not direct side effects.

## 4. The operation contract (define first — everything imports it)

```ts
// 1. What the shell / workflow / agent sends. A request, never a side effect.
interface OperationRequest {
  schema_version: 'scbe.operation.v1';   // versioned for forward contract changes
  op: string;            // namespaced verb: "llm.chat" | "browser.navigate" | "fs.read" | "terminal.command.request" | "metrics.read"
  args: Record<string, unknown>;
  request_id: string;    // uuid; idempotency key (mandatory)
  origin: { kind: 'app' | 'workflow' | 'agent'; id: string };
  workspace?: { id: string; root: string };  // logical id (preserved in audit) + root path (gate confines to it; DENY outside)
  privacy: 'local_only' | 'external_api';     // names the real risk: stays on-box vs leaves to a third party
  budget_cents?: number;
  dry_run?: boolean;     // preview/validate without executing the side effect
}

// 2. The gate's verdict — produced and audited BEFORE any handler runs.
interface OperationDecision {
  request_id: string;
  decision: 'ALLOW' | 'QUARANTINE' | 'ESCALATE' | 'DENY';   // L13 tiers
  zone: 'GREEN' | 'YELLOW' | 'RED';                          // HyperLane
  reason: string;
  policy: string;        // which rule fired
  latency_ms: number;
}

// 3. The handler's outcome — refs, not blobs.
interface OperationResult {
  request_id: string;
  ok: boolean;
  output?: Record<string, unknown>;             // op-specific; may be a ref/summary, not raw content
  error?: { code: string; message: string };
  artifacts?: { kind: string; ref: string }[];  // screenshot path, file ref — references, never inline blobs
  duration_ms: number;
}

// 4. The append-only audit row — written twice: at request, completed at result.
interface AuditRecord {
  schema_version: 'scbe.operation.v1';   // versioned for forward contract changes
  request_id: string;
  ts_request: string;                 // ISO
  op: string;
  origin: OperationRequest['origin'];
  privacy: string;
  decision: OperationDecision;        // BEFORE half
  ts_result?: string;                 // AFTER half ↓
  result_summary?: { ok: boolean; error_code?: string; artifact_refs?: string[]; output_shape?: string };
  // explicitly NO secrets, NO full file contents, NO raw prompts/keys
}
```

## 5. Governance gate & audit

```
app | workflow | agent
  → OperationRequest
  → backend POST /v1/op
  → gate.govern(req)                 # agent-bus envelope + HyperLane zone
       → write AuditRecord (request + decision)            [BEFORE]
  → if decision != ALLOW: return audited denial, stop
  → handler(req.args)                # the ONLY place a real side effect happens
  → OperationResult
  → complete AuditRecord (result_summary)                  [AFTER]
  → return (or stream over WS /v1/events)
```

- **Audit before AND after.** Before: request + decision. After: result summary
  (ok / error code / artifact refs / output shape). Never secrets or full file
  contents. Append-only, in the agent-bus artifact pattern (`artifacts/agent_bus/…`
  or `.scbe/`).
- The gate is the **single chokepoint**. A handler is unreachable except through
  `govern()`; this is enforced by the backend's structure, not by convention.

## 6. Operation namespace & boundaries (hard requirements)

| Op | Handler | Boundary |
|---|---|---|
| `llm.chat` / `llm.complete` | `provider_executor` (glm-5.1:cloud default) | streamed over WS; prompts not audited verbatim |
| `web.search` / `browser.navigate` / `browser.extract` / `browser.screenshot` | governed Playwright lane | **backend-driven browser, not an iframe.** Shell renders state/screenshots/events; the backend navigates |
| `fs.read` / `fs.write` / `fs.list` | OS fs handler | **confined to `workspace.root`.** Out-of-scope path → DENY. No arbitrary `C:\Users\issda` browsing until the gate supports explicit trust scopes |
| `terminal.command.request` | allowlisted command handler | **operation console, not raw shell by default.** request → govern → allowlisted handler → result |
| `terminal.shell.raw` | raw shell handler | **RED zone; requires ESCALATE / stronger policy.** Not available by default |
| `metrics.read` | system metrics handler | read-only |

## 7. Workflow engine (`packages/workflow-engine`)

Standalone, embeddable, and an agent-bus tool. Built strictly against an
**injected `OperationClient`** so it is publishable and testable without the
desktop or a live backend.

```ts
interface OperationClient { run(req: OperationRequest): Promise<OperationResult>; }

interface WorkflowSpec {
  id: string;
  trigger: { kind: 'manual' | 'event' | 'schedule'; match?: Record<string, unknown> };
  steps: {
    id: string;
    when?: string;                 // condition over prior step outputs / event
    op: string;                    // emitted as an OperationRequest
    args: Record<string, unknown>; // may reference prior step outputs
  }[];
}
```

**Execution pipeline (dry-run from day one):**

```
validate(spec)            # schema + op names + arg references resolve
  → preview(spec)         # the ordered list of operations it WOULD emit
  → dryRun(spec)          # run with dry_run:true through the gate (governed, no side effects)
  → execute(spec)         # run for real, each step a governed OperationRequest
```

- Rules are the simple case of a workflow: a single `trigger → when → op`.
- The engine never calls a URL or a handler directly — only `client.run(req)`.
- Agent-bus tool adapter exposes `validate/preview/dryRun/execute` so agents can
  compose and trigger workflows under the same governance.

## 8. Desktop shell (clean rebuild)

- **Window manager**, taskbar, desktop, draggable/resizable windows.
- **App registry / plugin interface:** each app is a component + a manifest
  declaring the operations it needs; the shell grants no capability the manifest
  doesn't list.
- **`BackendClient`:** the only path out; wraps `POST /v1/op` and the `WS` event
  stream. Apps import `BackendClient`, never `fs`/SDKs/`fetch`-to-arbitrary.
- **Security rules baked in (the anti-Kimi list):** no `eval` / `new Function`;
  code-runner apps execute in a sandboxed Worker/iframe, never the page context;
  any HTML render is sanitized; no `dangerouslySetInnerHTML` on remote content.

## 9. Data flow & determinism

- One request lifecycle (Section 5); streaming results (e.g. `llm.chat`) flow as
  WS events keyed by `request_id`.
- `dry_run` short-circuits the handler after the gate, returning the decision +
  the operation it *would* run — used by workflow preview and by cautious apps.
- `request_id` is the idempotency + audit + stream-correlation key throughout.

## 10. Phasing

- **Phase 0 — clean shell scaffold.** Window manager + app registry +
  `BackendClient`, security rules baked in. No real capabilities yet.
- **Layer 0 — backend spine.** FastAPI: operation registry + gate + audit +
  `/v1/op` + `WS /v1/events` + `/health`. One handler stub to prove the path.
- **Phase 1 — real apps, slices first** (Section 11). Chat, then Browser, then
  FileManager / Terminal / SystemMonitor, each behind the same gate.
- **Phase 2 — workflow engine + builder app.** Only after Phase 0/1 proves the
  operation registry.

**Warning (load-bearing): do not overbuild Phase 2 before Phase 0/1 proves the
registry.** The product succeeds or fails on whether one app can safely do one
real thing through the gate.

## 11. Vertical slices (the Phase-1 acceptance bar)

The bar stays exactly this narrow — five conditions, both slices:

1. `llm.chat` proves streamed governed model access.
2. `browser.navigate` proves governed external-world interaction.
3. Both write before/after audit rows.
4. Both support `dry_run`.
5. Neither may bypass `/v1/op`.

**Slice 1 — Chat (the spine-prover):**
```
Desktop Chat app
  → POST /v1/op  { op: "llm.chat", args: {messages}, ... }
  → gate.govern → OperationDecision (audited)
  → provider_executor (glm-5.1:cloud)
  → streamed result over WS /v1/events
  → AuditRecord completed
```
Acceptance: a real model reply renders in the Chat window; an audit row exists
with a before-decision and an after-summary; no secret/prompt logged verbatim.

**Slice 2 — Browser (proves the remote-control lane):**
```
Browser app
  → POST /v1/op  { op: "browser.navigate", args: {url}, ... }
  → gate.govern (HyperLane zone) → decision (audited)
  → governed Playwright worker navigates
  → screenshot/state event over WS
  → AuditRecord completed
```
Acceptance: shell shows a real screenshot/state from a backend-driven browser
(no iframe); a YELLOW/RED URL is DENY'd or ESCALATE'd and audited as such.

If both work, the rest of the desktop has a real spine.

## 12. Testing

- **Contract tests:** every handler round-trips `OperationRequest` →
  `OperationResult`; malformed requests rejected.
- **Gate tests:** out-of-`workspace` `fs` path → DENY; `terminal.shell.raw`
  without escalation → DENY/ESCALATE; YELLOW/RED web zone handled.
- **Audit tests:** a before-record and an after-record exist for every op; no
  secrets / full file contents present in any row.
- **Engine tests:** `validate`/`preview`/`dryRun` against a mock `OperationClient`
  (no backend); dry-run emits zero side effects; golden workflow → golden
  operation sequence.
- **Slice acceptance:** Section 11.

## 13. Location & repos

- **Desktop:** standalone product repo (e.g. `aether-desktop`), consuming
  published `scbe-agent-bus` + `workflow-engine`. Not dumped into SCBE core.
- **Engine:** `SCBE-AETHERMOORE/packages/workflow-engine/` (library code per
  `STORAGE_RULES`; publishable; the agent tool).
- **Secrets:** env / `.secrets/`, never committed.

## 14. Out of scope (YAGNI for now)

- Porting all ~80 Kimi apps. Games/utilities are reimplemented clean only as
  needed; the high-value apps (Chat, Browser, FileManager, Terminal,
  SystemMonitor, workflow-builder) lead.
- Multi-machine / remote-host desktops.
- Raw shell access by default (gated behind `terminal.shell.raw` + ESCALATE).
- App marketplace / third-party plugins beyond the internal registry.

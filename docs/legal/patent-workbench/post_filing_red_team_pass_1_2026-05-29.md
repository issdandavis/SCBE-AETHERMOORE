## Executive View

The filing's strongest posture is the concrete runtime machine: proposed action -> structured context vector -> bounded Poincare-ball state -> session centroid drift -> nonlinear governance cost -> allow/review/quarantine/deny/reroute -> audit or receipt artifact. This is the frame to preserve in any later response.

The highest-risk posture is over-describing the filing as "safe AI," "mathematical proof of alignment," or a broad ownership claim over hyperbolic embeddings. Those framings invite section 101 and prior-art attacks. The safer frame is a specific computer-executed control pipeline that changes whether a computational action runs.

## Claim Family Strength Tiers

### Strongest

- Claims 1-2: hyperbolic drift governance if argued as runtime execution control, not math alone.
- Claims 5, 8, 9: session centroid and durable runtime state; concrete machine-state continuity.
- Claims 10-11: quarantine containment and ordered pre-filter stack; clear operational behavior.
- Claims 15-20: bijective tamper/canonical AST/identifier checks; strong software-security mechanism.
- Claim 24: null-space anomaly in the live RuntimeGate path.
- Claims 26-27: disjoint serialized token alphabets and harmonic phase offsets, if kept technical.

### Medium / Review-Sensitive

- Claim 3: alternative nonlinear formulas. Needs careful explanation that these are species/embodiments, not all active in the same code path.
- Claim 12: PQC receipt. Good embodiment, but avoid making novelty depend on ML-KEM/ML-DSA themselves because those primitives are standardized prior art.
- Claim 21-22: Sacred Egg / predicate container. Useful, but needs support citations and should be argued as an authorization container embodiment.
- Claim 23: reroute. Useful but newer; watch priority support.
- Claim 25: fleet/task physics. Good breadth, but more vulnerable to "abstract scheduling metaphor" unless tied to actual task assignment state/cost.
- Claim 28: domain-specific entropy encoding. Needs careful support and should avoid sounding like pure information labeling.

## Competitor Design-Around Scenarios

1. Flat vector gate
   - Uses Euclidean cosine distance from a session centroid and routes actions.
   - Risk: may avoid hyperbolic-specific dependents but still close to claim 1 if claim construction reads "bounded hyperbolic space" strictly.
   - Prep: product value should show why hyperbolic distance changes outcomes, not just that it is different math.

2. Hyperbolic anomaly detector without execution control
   - Scores drift but does not directly allow/review/quarantine/deny execution.
   - Risk: less likely to infringe core execution-routing limitation.
   - Prep: keep execution control central in all public and prosecution language.

3. LLM guardrail stack without geometry
   - Regex/model/router filters only.
   - Risk: likely outside the core geometry claims.
   - Prep: our product can still compete via benchmark evidence, but patent moat is the geometry+runtime combination.

4. Geometry gate without Sacred Tongues/tokenizer
   - Uses hyperbolic drift and state, but no six-axis phi labels or SS1 alphabet.
   - Risk: independent claims may still read; dependents 4/26-28 do not.
   - Prep: do not over-anchor commercial positioning to Sacred Tongues only.

5. Tokenizer tamper detector without governance gate
   - Does AST round-trip tamper detection but only reports alerts.
   - Risk: claims 15-20 are stronger where tamper signal gates execution.
   - Prep: keep "blocks/escalates proposed computational action" language central.

# Pyramid Atlas Mapping

Date: 2026-04-11
Status: research-grounded design note

## Purpose

Ground the pyramid / hall / celestial-alignment ideas in evidence that is strong enough to use as a systems reference, then map that material into the existing SCBE stack without pretending fringe claims are established archaeology.

## Evidence split

### Solid enough to use as reference

- The Great Pyramid is aligned very closely to the cardinal directions.
- The Khufu pyramid has an internal circulation structure: descending passage, ascending passage, Grand Gallery, King's Chamber, Queen's Chamber, and relieving spaces.
- The so-called star shafts are real architectural features; astronomical interpretations exist and are plausible as orientation references, though exact target stars and ritual intent remain debated.
- The structure is useful as an example of layered navigation, constrained corridors, axial orientation, and nested access.

### Weak or speculative

- "The Great Pyramid had pumps" is not accepted archaeology.
- Hydraulic engineering papers exist for Egyptian pyramids, but the strongest recent hydraulic claim concerns Djoser's Step Pyramid, not Khufu's Great Pyramid.
- "Power plant" or literal machine-pump interpretations of the Great Pyramid should be treated as non-canonical for system design.

## What is transferable

The useful transfer is structural, not mystical:

- base platform
- chambers
- corridors
- shafts
- cardinal orientation
- external celestial reference
- hidden / void spaces that still affect traversal and interpretation

This is enough to support an atlas-style architecture:

- local rooms
- halls within a wing
- shafts / gateways between scales
- a fixed reference frame for orientation
- voids as meaningful absences, not just missing data

## Mapping to MemPalace

MemPalace already provides the symbolic topology:

- wing
- hall
- room
- closet
- drawer
- tunnel

The best fit is:

- pyramid base -> wing
- hall network -> MemPalace halls
- chamber -> room
- shaft -> zoom anchor / scale gateway
- tunnel -> cross-wing bridge
- hidden void -> null-space support deficit or latent bridge region

MemPalace should remain the symbolic retrieval substrate, not the manifold itself.

## Mapping to SCBE

The SCBE side should use the pyramid material as an orientation and traversal model:

- cardinal alignment -> global orientation frame
- star alignment -> external reference frame
- corridor geometry -> allowed transition lanes
- voids / hidden spaces -> null-space pressure
- layered internal access -> staged recall and staged execution

That yields a practical split:

- PumpPacket chooses the active state profile.
- MemPalace chooses symbolic locality.
- Zoom anchors move between local charts.
- The manifold scores coherence, support, and containment.

## Orientation frame

The pyramid material is most useful if reduced to a small orientation contract:

- `north_axis`: fixed directional basis
- `east_axis`: orthogonal local basis
- `celestial_anchor`: named external reference, if any
- `bearing_deg`: local heading
- `scale_band`: current chart scale

This lets the system describe transitions as oriented movement rather than generic jumps.

## Null-space interpretation

The "hidden space" analogy is valid only if stated carefully.

In this system, a void is not empty. It is one of:

- a missing support channel
- an unresolved bridge
- a latent chamber not yet surfaced by retrieval
- a structurally influential absence

That maps directly onto the null-space work already present in SCBE.

## Build implications

Use the pyramid reference to justify these concrete additions:

1. Zoom anchors as shaft-like scale gateways
2. Orientation frames on anchors
3. Hall-aware symbolic locators
4. Null-pressure metrics for latent or weakly connected regions
5. Retrieval and routing that prefer aligned corridors over arbitrary nearest-neighbor jumps

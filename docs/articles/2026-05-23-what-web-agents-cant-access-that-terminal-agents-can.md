# What Web Agents Can't Access That Terminal Agents Can (And Why Your Governance Layer Needs to Know the Difference)

Here's a claim that should seem obvious and mostly isn't enforced: a web-based AI agent cannot access `/home/user/documents`.

It doesn't matter what the agent says it can do. The browser sandbox doesn't allow it. There is no native filesystem access in a browser context unless the user has explicitly granted it through the File System Access API — and even then, access is scoped to specific directories, not the whole tree. An agent claiming `/home/user/documents` while running in a browser is either lying about what it has, confused about where it's running, or being used to probe for filesystem paths the attacker hopes you'll hand over.

Most governance systems don't check this. They watch for malicious *content* in requests, not structural impossibilities in the request's claimed context. The question "is this agent even capable of doing what it says it's doing given where it's running?" doesn't usually get asked.

We built something to ask it.

---

## The system-space model

SCBE's new operator system-space model extends the GeoSeal governance framework to cover the structural position of an operator in the system topology — not physical geography, but the layer of the stack they're operating from.

There are four access planes:

- **web** — browser context; OPFS (Origin Private File System) only; no native filesystem
- **terminal** — shell context; full filesystem access based on user permissions
- **app** — desktop/mobile sandbox; app-data directory only; no system paths
- **api** — HTTP caller; no direct filesystem access at all; server-side FS only

Each plane crossed with the operator's auth state produces a governance decision. The output is a ring (core/outer/restricted/blocked) that maps directly to the pipeline's L13 ALLOW/QUARANTINE/ESCALATE/DENY tiers, plus a deterministic space ID (a hash of plane+auth+session fingerprint), plus a derived filesystem topology.

The topology for a web-anonymous operator looks like this:

```json
{
  "sandboxLevel": "full",
  "accessibleRoots": [],
  "writablePaths": [],
  "persistsAcrossSessions": false,
  "tempOnly": true,
  "mountPoints": []
}
```

Empty arrays. No native paths. No mount points. Nothing to claim access to.

If that operator submits a request claiming access to `/home/user/documents`, the system raises a `CROSS_PLANE_CLAIM` flag, reduces trust by 0.30, and depending on the base trust level, may downgrade the ring from outer to restricted automatically.

## The specific thing that's new

The piece that isn't just "check the agent's context" is: this is deterministic and hash-anchored.

The space ID is computed as:

```
SHA256(plane | auth | session_fingerprint)[:16]
```

It's a 16-character hex string, stable across all calls with the same inputs. You can use it as a receipt anchor in an audit log — *this agent, at this system-space position, made this request*. The receipt is not self-reported by the agent; it's computed by the governance layer from the structural facts of the session.

That matters because agents lie. Not intentionally (usually), but because they're generated by models that don't always have accurate context about their own execution environment. An agent running via an API call might describe itself as having terminal access. The model generating its responses might not know it's behind an HTTP shim. The governance layer can't trust the agent's self-description; it has to derive the facts from the session structure.

## What the trust table looks like

| Plane | Auth State | Ring | Trust |
|-------|------------|------|-------|
| terminal | sudo | core | 0.95 |
| terminal | authenticated | core | 0.85 |
| terminal | service_account | core | 0.80 |
| terminal | anonymous | restricted | 0.35 |
| app | authenticated | outer | 0.70 |
| app | service_account | outer | 0.65 |
| app | anonymous | restricted | 0.25 |
| api | service_account | core | 0.80 |
| api | authenticated | outer | 0.70 |
| api | anonymous | blocked | 0.05 |
| web | authenticated | outer | 0.60 |
| web | anonymous | blocked | 0.05 |

Terminal sudo gets core ring (0.95) — highest trust, but also raises the `ELEVATED_TERMINAL` governance flag automatically. Elevated trust doesn't mean unmonitored. It means the geometry is calibrated for a context where real damage is structurally possible.

Web anonymous gets blocked (0.05) by default. Not because web agents are useless — because an unauthenticated browser caller with no session fingerprint and no login time has almost no structural basis for trust. If they try to claim filesystem paths, they're out.

---

## The penalty system

Two violations currently reduce trust at evaluation time:

- `CROSS_PLANE_CLAIM` — agent claims a path structurally impossible in their plane: **−0.30 trust**
- `SESSION_MISSING_LOGIN_TIME` — auth state says authenticated but no session timestamp provided: **−0.15 trust**

These penalties can stack. A terminal+authenticated agent (base trust 0.85) that claims an HTTP URL as a file path *and* provides no login timestamp ends up at 0.40 — right at the restricted/outer boundary. The ring downgrades from core to restricted automatically.

The threshold is `≤ 0.40` for restricted, `< 0.10` for blocked. An authenticated agent with stacked violations lands in restricted. An anonymous web agent with no claims doesn't need violations to hit blocked — the base trust is already 0.05.

The `ringToGovernanceTier()` function gives you the L13 vocabulary directly:

```typescript
ringToGovernanceTier('core')       // 'ALLOW'
ringToGovernanceTier('outer')      // 'QUARANTINE'
ringToGovernanceTier('restricted') // 'ESCALATE'
ringToGovernanceTier('blocked')    // 'DENY'
```

The operator space decision slots into the same decision layer that handles everything else in the pipeline. It's not a separate security check you have to integrate separately — it's a new dimension of the existing governance geometry.

---

The question "where is this agent running and what can it structurally do from there?" is one more geometric fact about an input. The pipeline measures it the same way it measures everything else: distance, phase deviation, trust score, ring assignment. The governance layer doesn't need to understand browsers vs terminals. It needs to know: is this agent's claim consistent with its position in the system?

Mostly, it's not.

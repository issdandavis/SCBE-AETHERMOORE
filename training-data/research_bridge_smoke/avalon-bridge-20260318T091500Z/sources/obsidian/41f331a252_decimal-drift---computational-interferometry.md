# Decimal Drift — Computational Interferometry

## Overview

Decimal Drift functions as a high-precision forensic instrument within the [[14-Layer Architecture|14-layer security pipeline]], effectively converting the pipeline into a **computational interferometer**. Instead of treating IEEE 754 floating-point rounding errors as noise, the system utilizes them as a cryptographic "watermark" or "fingerprint" of computational provenance.

It should be understood as a **full-stack accumulation signal**, not a four-layer trick. Decimal drift builds as data traverses the system, with tiny deviations compounding into a measurable cycle/frequency history. Some layers expose that history more clearly than others, but the drift belongs to the entire path.

This mechanism prevents attacks by distinguishing between:
- **Organic Data**: Legitimately traversed the authorized mathematical layers
- **Synthetic Data**: Generated by attackers, scripts, or offline calculations

## 1. Mechanism: 14-Stage Computational Interferometry

As a data vector traverses the pipeline, each layer introduces deterministic but complex transformations that leave specific floating-point artifacts in the least significant bits. Over repeated traversal and long-lived system operation, these accumulate into a path signature: a subtle error build-up and frequency build-up that can be tracked across cycles.

### Layer Accumulation

| Layer | Operation | Drift Source |
|-------|-----------|-------------|
| **Layer 3** (Weighted Transform) | $G^{1/2} \cdot x$ | Multiplication rounding errors |
| **Layer 4** (Poincaré Embedding) | $\tanh$ activation ($\|u\| < 1$) | Precision compression distribution |
| **Layer 7** (Phase Transform) | Möbius addition + rotation | Complex-number arithmetic drift |

These are observation hotspots, not the entire phenomenon. The broader rule is:

- every layer can contribute small deterministic perturbations
- layer transitions preserve and reshape prior perturbations
- later layers can amplify, mask, or reveal earlier drift
- the final signature is a history of traversal, not a single local defect

### The Drift Vector

Legitimate data accumulates a specific drift vector $d_{\text{drift}}$ characterized by the exact order of operations in the authorized code. This creates a **"numerical narrative"** of the data's journey through the pipeline.

One useful intuition is:

- `1.0000000000000001` is not interesting by itself
- `1.00000000000100` after prolonged governed traversal is interesting

The security value comes from the accumulation pattern, not from any one isolated decimal tail.

### Attack Detection

An attacker attempting to inject data will fail because:
1. **"Too perfect"** — Calculated analytically without the pipeline's rounding constraints
2. **Wrong rounding signature** — Calculated via a different math library with different FP behavior

## 4. Spectral Decimal Drift — The "Hard Wall"

Extends to [[Audio Axis|Layer 14 (Audio Axis)]] and the [[Symphonic Cipher]].

### The Challenge
An attacker might replicate Sacred Tongue command frequencies (e.g., 440Hz + 880Hz) using a standard sine wave generator.

### The Defense
The SCBE [[Harmonic Scaling Law|HarmonicSynthesizer]] applies specific DSP filters and overtone masks that leave **arithmetic artifacts** in frequency peaks that a standard `sin()` function cannot produce.

### The Result
Even if the audio sounds correct (right frequencies), the system **rejects it** if the frequency peaks lack the specific floating-point drift signature of the authorized synthesis engine.

> This forces the attacker to not only steal the key but also replicate the exact floating-point behavior of the generation engine.
